## A) BLOQUEADORES DE LANCAMENTO

### A1. OAuth callback sem validacao de state (CSRF)
- **Onde:** `src/app/[locale]/(auth)/callback/route.ts:7-11`
- **Problema:** O callback OAuth nao valida o parametro `state`. Um atacante pode forjar link de callback e forcar troca de sessao (CSRF).
- **Impacto real:** Account hijacking — atacante pode vincular sua conta OAuth a sessao da vitima.
- **Fix:** Armazenar `state` em cookie HttpOnly antes do redirect OAuth; validar no callback; retornar 400 se nao bater. Usar PKCE.
- **Esforco:** Medio
- **Auto-fix:** Nao (requer decisao de design)

### A2. Stripe checkout sem idempotency key — risco de cobranca dupla
- **Onde:** `src/app/api/payment/create-intent/route.ts:86-98` e `src/app/api/bookings/checkout/route.ts:175-181`
- **Problema:** Nenhuma chamada `stripe.paymentIntents.create()` ou `stripe.checkout.sessions.create()` usa `idempotency_key`. Retry do cliente cria PaymentIntent duplicado.
- **Impacto real:** Cliente cobrado 2x. Perda financeira direta.
- **Fix:** Gerar idempotency_key deterministico (professional_id + service_id + timestamp). Passar para Stripe SDK.
- **Esforco:** Baixo
- **Auto-fix:** Sim

### A3. Triggers de banco referenciam coluna inexistente `booking_time`
- **Onde:** `supabase/migrations/20250217000000_sprint7_automations.sql:451,617,633`
- **Problema:** Triggers usam `NEW.booking_time` mas a coluna real e `start_time`. Trigger falha silenciosamente em todo UPDATE de booking.
- **Impacto real:** Automacoes de booking (loyalty, notificacoes via trigger) nunca disparam.
- **Fix:** Migration corretiva: ALTER trigger para usar `NEW.start_time`.
- **Esforco:** Baixo
- **Auto-fix:** Sim

### A4. Migrations com timestamp duplicado — ordem de execucao imprevisivel
- **Onde:** 6 arquivos com prefixo `20250218000001_*.sql`
- **Problema:** Supabase executa migrations em ordem alfabetica. Multiplas migrations com mesmo timestamp podem executar em ordem errada, causando falha (tabela referenciada antes de criada).
- **Impacto real:** Deploy quebra em ambiente limpo. Schema inconsistente.
- **Fix:** Renomear para timestamps sequenciais (000001, 000002, 000003...).
- **Esforco:** Baixo (rename + testar em ambiente limpo)
- **Auto-fix:** Sim

### A5. Zero error boundaries — crash de 1 componente derruba toda a pagina
- **Onde:** Nenhum `error.tsx` em `(dashboard)/`, `(auth)/`, `(public)/`, `(admin)/`
- **Problema:** Next.js App Router usa `error.tsx` como error boundary. Sem ele, qualquer throw em componente filho mostra tela branca.
- **Impacto real:** Erro em 1 componente do dashboard = dashboard inteiro inacessivel. Usuario perde confianca.
- **Fix:** Criar `error.tsx` em cada layout group com UI de fallback.
- **Esforco:** Baixo
- **Auto-fix:** Sim

## B) PROBLEMAS CRITICOS

### B1. Race condition no payment linking (fire-and-forget)
- **Onde:** `src/app/api/bookings/checkout/route.ts:193-199`
- **Problema:** Bloco async fire-and-forget atualiza tabela `payments` APOS booking criado. Se webhook Stripe chega antes, payment fica orfao.
- **Impacto real:** Payment no banco sem booking vinculado. Reconciliacao financeira falha.
- **Fix:** Usar transacao ou aguardar promise antes de retornar response.
- **Esforco:** Medio
- **Auto-fix:** Nao

### B2. Bot rate limiter existe mas NUNCA e chamado
- **Onde:** `src/lib/whatsapp/rate-limiter.ts` (definido) vs `src/lib/whatsapp/processor.ts` (nunca importa)
- **Problema:** Classe `WhatsAppRateLimiter` implementada mas nao integrada no fluxo de processamento.
- **Impacto real:** Usuario malicioso envia 1000 msgs/s → esgota quota Claude API → DoS para todos os profissionais.
- **Fix:** Chamar `RateLimiter.check()` em `processWhatsAppMessage()` antes de processar.
- **Esforco:** Baixo
- **Auto-fix:** Sim

### B3. Evolution API keys vazam em mensagens de erro
- **Onde:** `src/lib/whatsapp/evolution.ts:81-84`
- **Problema:** `throw new Error(\`Evolution API error: ${response.status} - ${error}\`)` — se Evolution retorna "Invalid API key: abc123", a key vaza no log.
- **Impacto real:** Credenciais em logs de servidor, error tracking (Sentry), ou ate resposta ao cliente.
- **Fix:** Sanitizar: logar apenas status code, nunca response body completo.
- **Esforco:** Baixo
- **Auto-fix:** Sim

### B4. bookings.service_id nullable — permite booking sem servico
- **Onde:** `supabase/migrations/20250101000000_init.sql` — coluna `service_id`
- **Problema:** FK para `services(id)` nao tem `NOT NULL`. Booking pode existir sem servico vinculado.
- **Impacto real:** Queries de analytics, receita, duracao retornam NULL. UI mostra "servico indefinido".
- **Fix:** `ALTER TABLE bookings ALTER COLUMN service_id SET NOT NULL;`
- **Esforco:** Baixo (verificar dados existentes antes)
- **Auto-fix:** Sim (com verificacao)

### B5. Sem validacao de amount antes de cobrar no Stripe
- **Onde:** `src/app/api/bookings/checkout/route.ts:175-181`
- **Problema:** Nao valida se `depositAmount > 0`, se `depositAmount <= servicePrice`, ou se excede limites Stripe.
- **Impacto real:** Profissional misconfigura deposito → cliente cobrado EUR 0.00 (erro Stripe) ou EUR 999999.
- **Fix:** Bounds check antes de criar session: `if (amount <= 0 || amount > 100000) return 400`.
- **Esforco:** Baixo
- **Auto-fix:** Sim

### B6. Cron send-reminders marca `reminder_sent=true` mesmo quando envio falha
- **Onde:** `src/app/api/cron/send-reminders/route.ts:193-199`
- **Problema:** Flag atualizada ANTES de confirmar envio via Evolution API. Se API falha, reminder nunca e reenviado.
- **Impacto real:** Cliente nao recebe lembrete → no-show → profissional perde receita.
- **Fix:** So marcar `reminder_sent=true` apos resposta 200 da Evolution API.
- **Esforco:** Baixo
- **Auto-fix:** Sim

### B7. reschedule_tokens com RLS `USING (true)` — qualquer user le qualquer token
- **Onde:** `supabase/migrations/20250217000000_sprint7_automations.sql:267`
- **Problema:** Policy `SELECT USING (true)` permite ler TODOS os tokens de reagendamento.
- **Impacto real:** Atacante com acesso anon pode listar tokens e reagendar bookings de outros.
- **Fix:** Remover policy publica ou adicionar filtro por token no USING.
- **Esforco:** Baixo
- **Auto-fix:** Sim

### B8. Incomplete logout — cookies Supabase podem sobreviver ao signOut
- **Onde:** `src/app/api/auth/signout/route.ts:11-13`
- **Problema:** Deleta TODOS os cookies genericamente, mas pode nao cobrir HttpOnly cookies do Supabase SSR.
- **Impacto real:** Sessao permanece ativa apos logout. Se browser compartilhado, proxima pessoa tem acesso.
- **Fix:** Chamar `supabase.auth.signOut()` PRIMEIRO, depois deletar cookies especificos (sb-access-token, sb-refresh-token).
- **Esforco:** Baixo
- **Auto-fix:** Sim

## C) MELHORIAS IMPORTANTES

### C1. Path traversal em gallery upload
- **Onde:** `src/app/api/gallery/upload/route.ts:47-48`
- **Problema:** `file.name` usado na construcao do path sem sanitizacao. Nome `../../etc/passwd` pode escapar do bucket.
- **Impacto real:** Baixo no Supabase Storage (sandboxed), mas violacao de principio.
- **Fix:** `const safeName = file.name.replace(/[^a-zA-Z0-9._-]/g, '_');`
- **Esforco:** Baixo | **Auto-fix:** Sim

### C2. 282 console statements em codigo de producao
- **Onde:** 95 arquivos em `src/`
- **Problema:** `console.log`, `console.error`, `console.warn` em routes e componentes. Vazam info sensivel (phones, URLs, tokens).
- **Fix:** Usar logger estruturado com niveis (debug/info/warn/error). Remover ou wrappear em NODE_ENV check.
- **Esforco:** Medio | **Auto-fix:** Parcial

### C3. 6 componentes com strings PT-BR hardcoded (i18n incompleto)
- **Onde:** `booking-form.tsx`, `time-slots.tsx`, `welcome-modal.tsx`, `qr-code-download.tsx`, `share-link-card.tsx`, `section-testimonials.tsx`
- **Problema:** Textos em portugues que nao passam pelo sistema i18n. Usuarios EN-US/ES-ES veem portugues.
- **Fix:** Extrair para namespaces de traducao.
- **Esforco:** Medio | **Auto-fix:** Sim

### C4. 16+ type assertions `as any` mascarando bugs
- **Onde:** `section-renderer.tsx` (6x), `settings-manager.tsx`, `my-page-editor.tsx`, `revenue-chart.tsx`, `booking-section.tsx`, `slug/page.tsx` (2x)
- **Problema:** `as any` esconde campos faltantes no tipo `Professional` e `PageSection`. Bugs so aparecem em runtime.
- **Fix:** Atualizar tipos em `types/database.ts` para incluir campos reais.
- **Esforco:** Medio | **Auto-fix:** Parcial

### C5. Memory leak no google-map.tsx
- **Onde:** `src/components/google-map.tsx:35-43`
- **Problema:** `setTimeout` de 10s nao e limpo se componente desmonta antes.
- **Fix:** `const timeoutId = setTimeout(...); return () => clearTimeout(timeoutId);`
- **Esforco:** Baixo | **Auto-fix:** Sim

### C6. Sem timeout em chamadas Claude API
- **Onde:** `src/lib/ai/chatbot.ts:297-303`
- **Problema:** SDK Anthropic sem `timeout` explicito. Se API trava, request fica preso ate timeout do Vercel (30s).
- **Fix:** Adicionar `timeout: 25_000` na chamada `.create()`.
- **Esforco:** Baixo | **Auto-fix:** Sim

### C7. Missing index em professionals.user_id (gargalo RLS)
- **Onde:** `supabase/migrations/20250101000000_init.sql`
- **Problema:** Toda policy RLS faz subquery `WHERE user_id = auth.uid()`. Sem indice, scan completo em toda request autenticada.
- **Fix:** `CREATE INDEX idx_professionals_user_id ON professionals(user_id);`
- **Esforco:** Baixo | **Auto-fix:** Sim

### C8. Dead code Meta Business API em crons
- **Onde:** `src/app/api/cron/send-reminders/route.ts:208-233`
- **Problema:** Bloco que trata provider "meta" — mas Meta API foi removida completamente do projeto.
- **Fix:** Deletar bloco morto.
- **Esforco:** Baixo | **Auto-fix:** Sim

### C9. DST bug no calculo de dia da semana
- **Onde:** `src/app/api/available-slots/route.ts:64`
- **Problema:** `new Date(date + 'T12:00:00')` pode cruzar fronteira DST. Data "2026-03-29" na Europa retorna dia errado.
- **Fix:** Usar `new Date(date + 'T00:00:00Z')` com UTC.
- **Esforco:** Baixo | **Auto-fix:** Sim

### C10. RPC cleanup_expired_tokens duplicada
- **Onde:** `src/app/api/cron/cleanup-tokens/route.ts:21,28`
- **Problema:** Chama `rpc('cleanup_expired_tokens')` 2x. Segunda chamada nunca executa (return anterior).
- **Fix:** Remover chamada duplicada.
- **Esforco:** Baixo | **Auto-fix:** Sim

### C11. Prompt injection via bot config (business name)
- **Onde:** `src/lib/ai/chatbot.ts:200+`
- **Problema:** System prompt construido dinamicamente com `businessInfo` do profissional. Se nome contem instrucoes maliciosas, Claude pode obedecer.
- **Fix:** Sanitizar businessInfo antes de injetar no prompt. Usar delimitadores claros.
- **Esforco:** Baixo | **Auto-fix:** Sim

### C12. Accessibility: aria-labels ausentes em 70+ elementos interativos
- **Onde:** Praticamente todos os componentes em `src/components/`
- **Problema:** Apenas 9 de 80 componentes usam `aria-label` ou `aria-required`. Nao atende WCAG 2.1 AA.
- **Fix:** Audit sistematico e adicao de labels.
- **Esforco:** Alto | **Auto-fix:** Parcial
