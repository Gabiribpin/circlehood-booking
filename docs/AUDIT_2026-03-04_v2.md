## BLOCKER (2)

### B1. WhatsApp webhook sem validação de assinatura
- **Arquivo**: `src/app/api/whatsapp/webhook/route.ts:51`
- **Impacto**: Atacante pode forjar mensagens WhatsApp, injetar prompts no Claude, criar agendamentos falsos
- **Evidência**: POST handler aceita qualquer JSON sem verificar header de assinatura
- **Correção**: Validar `x-signature` header com HMAC do body usando secret da Evolution API
- **Esforço**: Baixo (1h)

### B2. GitHub PAT exposto no client-side
- **Arquivo**: `src/app/[locale]/(admin)/admin/execution-wheel/page.tsx:37`
- **Evidência**: `process.env.NEXT_PUBLIC_GH_ISSUES_PAT` — prefixo NEXT_PUBLIC_ envia para o browser
- **Impacto**: Token GitHub visível no DevTools, permite criar/editar/fechar issues no repositório
- **Correção**: Remover prefixo NEXT_PUBLIC_, criar API route server-side para proxy das chamadas GitHub
- **Esforço**: Médio (2-3h)

## CRITICAL (4)

### C1. Stripe checkout usa client-side Stripe.js no servidor
- **Arquivo**: `src/app/api/stripe/checkout/route.ts:3`
- **Evidência**: `import { getStripe } from '@/lib/stripe'` — `getStripe()` retorna `loadStripe()` (browser-only)
- **Impacto**: Criação de checkout sessions falha em runtime (Stripe.js não funciona no server)
- **Correção**: Trocar para `import { getStripeServer } from '@/lib/stripe/server'`
- **Esforço**: Baixo (15min)

### C2. CSV import vulnerável a parsing malicioso
- **Arquivo**: `src/app/api/contacts/import/route.ts:41`
- **Evidência**: `line.split(',')` — não trata campos com aspas, vírgulas internas, ou newlines
- **Impacto**: Dados desalinhados, campos escritos em colunas erradas
- **Correção**: Usar biblioteca CSV robusta (papaparse ou csv-parse)
- **Esforço**: Baixo (1h)

### C3. Race condition no checkout de pagamento
- **Arquivo**: `src/app/api/bookings/checkout/route.ts:148-200`
- **Evidência**: Booking inserido ANTES da sessão Stripe ser criada. Se Stripe falhar, booking fica em `pending_payment` eternamente
- **Impacto**: Slots bloqueados por bookings órfãos (até o cron expire-pending rodar)
- **Correção**: Criar sessão Stripe primeiro, depois inserir booking; ou usar transaction
- **Esforço**: Médio (2h)

### C4. Admin password reutilizada como HMAC secret
- **Arquivo**: `src/lib/admin/session.ts:13,36`
- **Evidência**: `createHmac('sha256', process.env.ADMIN_PASSWORD)` — mesma secret para auth e signing
- **Impacto**: Se password for fraca, tokens de sessão são facilmente forjáveis
- **Correção**: Usar ADMIN_SESSION_SECRET separado, derivar com HKDF
- **Esforço**: Baixo (30min)

## HIGH (8)

### H1. Gallery upload sem validação de MIME type
- **Arquivo**: `src/app/api/gallery/upload/route.ts:63`
- **Evidência**: `safeExt()` só sanitiza extensão, aceita qualquer tipo de arquivo (exe, php, svg com JS)
- **Impacto**: Upload de arquivos maliciosos, potencial XSS via SVG
- **Correção**: Whitelist de MIME types (`image/jpeg`, `image/png`, `image/webp`) + validação de magic bytes
- **Esforço**: Baixo (1h)

### H2. Rate limiting em memória (admin login)
- **Arquivo**: `src/lib/admin/session.ts:66-88`
- **Evidência**: `new Map<string, { count, resetAt }>()` — reseta a cada deploy
- **Impacto**: Brute force do admin password possível em deployments distribuídos
- **Correção**: Usar Redis ou rate limiting no edge (Vercel)
- **Esforço**: Médio (3h)

### H3. Sem rate limiting no endpoint público de bookings
- **Arquivo**: `src/app/api/bookings/route.ts`
- **Evidência**: Nenhum mecanismo de rate limiting — apenas idempotência de 5min
- **Impacto**: Spam de bookings falsos, flood de notificações email/WhatsApp
- **Correção**: Rate limit por IP ou por professional_id (10/hora)
- **Esforço**: Médio (2h)

### H4. Índices faltando em FKs usadas por RLS
- **Arquivos**: Migrations de `whatsapp_config`, `ai_instructions`, `whatsapp_templates`, `notifications`
- **Evidência**: Coluna `user_id` usada em subqueries RLS sem índice
- **Impacto**: Full table scan em CADA request autenticado, degradação com crescimento
- **Correção**: `CREATE INDEX` nas colunas `user_id` dessas tabelas
- **Esforço**: Baixo (30min)

### H5. Sessão admin sem revogação server-side
- **Arquivo**: `src/lib/admin/session.ts:12-24`
- **Evidência**: Tokens auto-validados (HMAC), sem storage server-side
- **Impacto**: Impossível revogar tokens, detectar roubo, ou limitar sessões
- **Correção**: Armazenar sessions no banco, validar contra DB
- **Esforço**: Alto (4h)

### H6. Stripe Connect sem verificar charges_enabled
- **Arquivo**: `src/app/api/bookings/checkout/route.ts:81-90`
- **Evidência**: Só verifica `startsWith('acct_')`, não verifica se conta pode receber pagamentos
- **Impacto**: Cobranças criadas para contas com onboarding incompleto, fundos ficam presos
- **Correção**: Verificar `charges_enabled` e `payouts_enabled` via Stripe API ou campo no banco
- **Esforço**: Médio (2h)

### H7. Webhook Stripe sem idempotência
- **Arquivo**: `src/app/api/webhooks/stripe-deposit/route.ts:29-34`
- **Evidência**: Sem dedup de event ID; Stripe pode reenviar webhooks se timeout
- **Impacto**: Booking confirmado 2x, notificações duplicadas
- **Correção**: Armazenar `event.id` processados ou usar WHERE com status atual
- **Esforço**: Médio (2h)

### H8. Supabase remote patterns muito permissivos
- **Arquivo**: `next.config.ts:32-41`
- **Evidência**: `hostname: '*.supabase.co'` — aceita qualquer subdomínio Supabase
- **Impacto**: Imagens de projetos Supabase maliciosos podem ser carregadas
- **Correção**: Usar hostname específico do projeto
- **Esforço**: Baixo (15min)

## MEDIUM (6)

### M1. CSP com unsafe-inline e unsafe-eval
- **Arquivo**: `next.config.ts:8-9`
- **Evidência**: `script-src 'unsafe-inline' 'unsafe-eval'` — necessário para Next.js hydration
- **Impacto**: Reduz proteção XSS (trade-off conhecido do Next.js)
- **Correção**: Migrar para nonce-based CSP quando Next.js suportar nativamente
- **Esforço**: Alto (requer mudança de framework)

### M2. N+1 queries nos endpoints de analytics
- **Arquivos**: `src/app/api/analytics/clients/route.ts`, `revenue/route.ts`, `services/ranking/route.ts`
- **Evidência**: Query separada para bookings e depois para services
- **Impacto**: 2 roundtrips por request, lentidão com muitos dados
- **Correção**: Usar Supabase JOINs (`.select('*, services(price)')`)
- **Esforço**: Baixo (1h)

### M3. Strings hardcoded em português em componentes públicos
- **Arquivos**: `src/components/google-map.tsx:47,146`, `src/components/public-page/section-gallery.tsx:44,78`
- **Evidência**: `'Carregando mapa...'`, `'Todos'` — não usam i18n
- **Impacto**: Usuários en-US e es-ES veem texto em português
- **Correção**: Usar `useTranslations()` com chaves no namespace correto
- **Esforço**: Baixo (30min)

### M4. Stripe error message expõe detalhes internos
- **Arquivo**: `src/app/api/stripe/connect/create-account/route.ts:109-111`
- **Evidência**: `{ error: '...', detail: message }` — message pode conter info da API Stripe
- **Impacto**: Vazamento de informação interna para atacante
- **Correção**: Logar detail server-side, retornar mensagem genérica
- **Esforço**: Baixo (15min)

### M5. Sem error boundaries nas rotas do dashboard
- **Arquivos**: Nenhum `error.tsx` encontrado nas rotas do dashboard
- **Impacto**: Erros não tratados crasham a página inteira
- **Correção**: Criar `error.tsx` no root de cada layout segment
- **Esforço**: Baixo (1h)

### M6. Webhook Stripe sem validação de timestamp
- **Arquivo**: `src/app/api/webhooks/stripe-deposit/route.ts`
- **Evidência**: `constructEvent` valida assinatura mas código não verifica `event.created`
- **Impacto**: Replay attacks com eventos antigos
- **Correção**: Rejeitar eventos com mais de 5min de idade
- **Esforço**: Baixo (15min)
