## NOVAS FINDINGS v2 — HIGH (P2)

### H-01v2: Payment create-intent usa rate limit in-memory (bypass em serverless) — [#342](https://github.com/Gabiribpin/circlehood-booking/issues/342)

**Severidade:** HIGH
**Arquivo:** `src/app/api/payment/create-intent/route.ts:9-23`
**Problema:** Rate limiting usa `Map()` local em vez do módulo Redis (`@/lib/rate-limit`). Em ambiente serverless, cada cold start cria novo Map — atacante pode bypass fazendo requests distribuídos.
**Impacto:** Spam de payment intents no Stripe, potencial custo financeiro.
**Fix:** Substituir `rateLimitMap` local por `isRateLimited()` de `@/lib/rate-limit`.
**Esforço:** 15min

### H-02v2: Slug collision race condition em complete-profile — [#343](https://github.com/Gabiribpin/circlehood-booking/issues/343)

**Severidade:** HIGH
**Arquivo:** `src/app/api/auth/complete-profile/route.ts:48-74`
**Problema:** Check de slug duplicado (`.maybeSingle()`) seguido de insert tem janela de race condition. Dois registros simultâneos com mesmo slug podem ambos passar o check.
**Impacto:** Slug duplicado no banco, booking page incorreta.
**Fix:** Usar `ON CONFLICT` no upsert ou tratar 23505 (unique_violation) com retry.
**Esforço:** 30min

## NOVAS FINDINGS v2 — MEDIUM (P3)

### M-01v2: contacts/import-whatsapp sem rate limit — [#344](https://github.com/Gabiribpin/circlehood-booking/issues/344)

**Severidade:** MEDIUM
**Arquivo:** `src/app/api/contacts/import-whatsapp/route.ts:22-217`
**Problema:** Endpoint chama Evolution API externamente sem rate limit. Com `maxDuration=60`, pode ser usado para resource exhaustion.
**Fix:** Adicionar `isRateLimited()` — 1 import por 5 minutos por professional.
**Esforço:** 15min

### M-02v2: Evolution API instance name interpolado em URL sem whitelist — [#345](https://github.com/Gabiribpin/circlehood-booking/issues/345)

**Severidade:** MEDIUM
**Arquivo:** `src/app/api/evolution/get-qrcode/route.ts:15-19`
**Problema:** Instance name validado por regex mas interpolado direto na URL do Evolution API. Sem verificação que instance pertence ao usuário autenticado.
**Fix:** Validar que instance corresponde ao whatsapp_config do user.
**Esforço:** 30min

### M-03v2: page-sections bulk PUT sem atomicidade — [#346](https://github.com/Gabiribpin/circlehood-booking/issues/346)

**Severidade:** MEDIUM
**Arquivo:** `src/app/api/page-sections/route.ts:258-270`
**Problema:** PUT atualiza até 20 seções em `Promise.all` sem transação. Requests concorrentes podem causar ordenação inconsistente.
**Fix:** Envolver em `supabase.rpc()` transacional ou usar versioning.
**Esforço:** 1h
